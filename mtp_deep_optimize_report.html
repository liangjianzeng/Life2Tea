💡 分析与建议

🚨 MTP 未生效或效率低下

测试显示 禁用 MTP 反而更快 (11.8 vs 9.4 tok/s)，这可能意味着：

  • 模型文件问题：Qwen3.6-27B-Q4_K_M-mtp.gguf 可能不是真正的 MTP 模型
  • llama.cpp 实现问题：MTP 推测解码的开销可能超过了收益
  • 建议：尝试不使用 -mtp 后缀的模型，或者检查模型是否正确支持 MTP

🔧 传统优化参数无效

测试了多种优化参数，但 都没有明显效果：

  • 调整线程数 (8, 16, 32)：速度稳定在 9.2-9.4 tok/s
  • 禁用 KV cache 量化：无效果
  • 使用 --no-mmap：无效果
  • 结论：瓶颈在模型规模和内存带宽，不在 CPU 线程或 KV cache

📉 性能瓶颈分析

当前速度 (9-12 tok/s) 较慢的原因：

  • 模型规模：27B 参数需要大量计算
  • 内存带宽：GB10 的 LPDDR5X (~273 GB/s) 是主要瓶颈
  • 量化精度：Q4_K_M 在精度和速度间取得平衡，但仍有优化空间

🚀 进一步优化建议

要突破 15 tok/s，可以尝试：

  • 1. 使用更小的模型：尝试 Qwen3.6-14B 或 Qwen3.6-8B
  • 2. 更激进的量化：使用 Q3_K_M 或 Q2_K (会损失精度)
  • 3. 检查 GPU 频率：确保 GB10 的 GPU 运行在最高频率
  • 4. 使用 vLLM：远程服务器上已经启动了 vLLM (端口 8100)，可以测试 vLLM 的性能
  • 5. 正确的 MTP 模型：确保使用真正支持 MTP 的模型（需要特殊训练）
