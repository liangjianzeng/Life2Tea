🐢 llama.cpp (慢)

  • 模型: Qwen3.6-27B (稠密)
  • 活跃参数: 27B (100%)
  • 量化: Q4_K_M (4-bit)
  • 速度: 11.8 tok/s
  • TTFT: 141 ms
  • 优点: 易用、支持更多模型格式
  • 缺点: 速度慢、MoE 支持差

🚀 vLLM (快)

  • 模型: Qwen3.6-35B-A3B (MoE)
  • 活跃参数: 3B (~11%)
  • 量化: FP8 (8-bit)
  • 速度: 52.2 tok/s
  • TTFT: 70 ms
  • 优点: 速度快、内存高效
  • 缺点: 需要更多依赖、配置复杂

💡 为什么 vLLM 快这么多？

1️⃣ MoE 架构优势

Mixture of Experts (MoE) 是关键技术：

  • 稠密模型 (llama.cpp): 每次前向传播要计算 全部 27B 参数
  • MoE 模型 (vLLM): 每次只激活 3B 参数 (一个 token 只路由到少数几个 expert)
  • 计算量相差 9 倍！这就是为什么 MoE 模型推理快
  • Qwen3.6-35B 有 35B 总参数，但每次只用 3B

2️⃣ vLLM 推理优化

vLLM 是专门为 LLM 推理优化的引擎：

  • PagedAttention: 高效管理 KV Cache 内存（类似操作系统的虚拟内存）
  • Continuous Batching: 动态批处理，提高 GPU 利用率
  • FlashAttention: 快速注意力计算（已集成到 vLLM）
  • FP8 量化: 比 llama.cpp 的 Q4_K_M 更高效

3️⃣ TTFT 大幅降低

首 Token 延迟 (TTFT) 对用户体验至关重要：

  • llama.cpp: 141 ms
  • vLLM: 70 ms (快 2 倍)
  • 这意味着 vLLM 的响应更即时，用户体验更好

4️⃣ MTP 未生效的原因

之前的测试发现 MTP 反而降低性能，可能原因：

  • Qwen3.6-27B-Q4_K_M-mtp.gguf 可能不是真正的 MTP 模型
  • MTP 需要模型在训练时特殊设计（预测多个未来 token）
  • llama.cpp 的 MTP 实现可能还不成熟
  • 建议: 使用 MoE 模型 + vLLM，比 MTP 更有效

📝 在 Life2Tea 中使用 vLLM

要将 vLLM 集成到 Life2Tea，需要：

  • 1. 添加 vLLM 后端: 在 backend/app/routers/ 中添加 vLLM router
  • 2. 配置模型: 在家长管理中添加 vLLM 模型配置
  • 3. 负载均衡: 如果有多个 vLLM 实例，可以添加负载均衡
  • 4. 监控: 添加 vLLM 性能监控（GPU 利用率、请求队列等）

🚀 进一步优化

要突破 100 tok/s，可以尝试：

  • 1. 使用更小的 MoE 模型: 如 Qwen3.6-14B-A2B (2B active)
  • 2. 量化优化: 尝试 INT4 或更低精度量化
  • 3. 多 GPU: 如果有多个 GB10，可以使用 tensor parallelism
  • 4. 批处理: 增大 batch size 提高吞吐量（但可能增加 TTFT）
